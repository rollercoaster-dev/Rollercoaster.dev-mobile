# i18n: migrate Focus Mode screen (issue #69)

Date: 2026-05-23
Issue: https://github.com/rollercoaster-dev/Rollercoaster.dev-mobile/issues/69
Milestone: native-rd: i18n

## Pre-flight findings

- The `focusMode` namespace already exists in `apps/native-rd/src/i18n/resources/en/focusMode.json` (as an empty `{}`), wired into `i18n/index.ts` and `i18next.d.ts`. This is a fill-in task, not greenfield.
- `StepCard` is already migrated (reads from `common.stepCard.*` and `common.evidenceTypes.*`, populated by #65). Its strings are out of scope.
- The issue's "33 `Alert.alert` matches per scan" is stale. `FocusModeScreen.tsx` contains exactly **one** `Alert.alert` call (line 471, "Could not delete evidence"). The other 32 were counted in a broader sweep that has since been refactored. Will note this on the issue and in the PR body.
- `dep:blocked` label is stale: foundation (#67) and shared labels (#65) are closed. testIDs (#64) is open but not blocking — current FocusMode tests query by text/labels and that pattern stays after migration.

## Scope decision

**Narrow PR matching the issue text strictly**, plus one carved-off sibling issue.

This PR (#69):

- `FocusModeScreen.tsx` — screen-owned copy → `focusMode.*`
- `FocusPillTabBar.tsx` — app-chrome copy → `common.navigation.*`

Sibling issue (filed: #131):

- "i18n: migrate shared focus widgets — GoalEvidenceCard, EvidenceDrawer, MiniTimeline, ProgressDots, ModeIndicator → `common.*`"

Why split: these widgets render across Focus/Goals/Capture/Completion. Doing them under #69 would force everyone working in parallel on #68/#70/#71/#72 to rebase. The sibling should land before #68 to avoid migration churn.

Acceptance criterion #1 ("No raw display strings in Focus Mode surfaces") is read as "no raw strings _owned by_ Focus Mode surfaces" — the shared widgets are tracked under the sibling.

## Implementation steps

1. **File the sibling issue** for shared widgets (GoalEvidenceCard, EvidenceDrawer, MiniTimeline, ProgressDots, ModeIndicator). Body lists the widgets, the raw strings, and notes it should land before #68.
2. **Branch**: stay on the current worktree branch (it already targets #69); rename trailer at PR time if needed.
3. **Commit 1 — populate `focusMode.json`**: fill `en/focusMode.json` with the keys above.
4. **Commit 2 — extend `common.json`**: add `navigation.tabs.{goals,badges,settings}`, `navigation.fab.newGoal`, `actions.undo`.
5. **Commit 3 — migrate `FocusModeScreen.tsx`**: replace 18 raw strings with `t()` calls. Hook already has `useTranslation`; pass interpolation values for the 4 parameterised strings.
6. **Commit 4 — migrate `FocusPillTabBar.tsx`**: add `useTranslation`, replace `TAB_LABELS` constant with a `t()`-derived lookup inside the render, swap FAB a11y label.
7. **Commit 5 — regenerate pseudo locale**: `bun run gen:pseudo` (auto-discovers populated files). Commit the regenerated `pseudo/focusMode.json` + updated `pseudo/common.json`.
8. **Verify**:
   - `bun run type-check`
   - `bun test --testPathPatterns "FocusMode|FocusPillTabBar"`
   - Tests keep their English assertions (default `lng:"en"`); only touch them if a string was reworded (none planned).
   - Manual: toggle pseudo locale in Settings dev section and confirm every Focus Mode surface string is bracketed.
9. **PR**:
   - Title: `feat(native-rd): i18n migrate Focus Mode screen (closes #69)`
   - Body notes the stale "33 Alert.alert" count and links the sibling issue.
   - Body explicitly lists what's deferred to the sibling.
10. **Issue comment on #69**: short note explaining the actual `Alert.alert` count is 1 (not 33) so future readers don't chase ghosts.
