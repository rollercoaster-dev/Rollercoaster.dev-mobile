# Development Plan: Issue #411

## Issue Summary

**Title**: [Storybook] Badge Detail — evidence "proof spine"
**Type**: feature (presentational component + a new design-token color group, Storybook-only)
**Complexity**: MEDIUM (see LOC/split note in Decisions D0)
**Estimated Lines**: ~430–480 lines total (component ~260 + new `evidence-*` token group, sized against the 7 product themes, not 14 + adapter/variants/compose wiring + contrast tests ~170–210). **This sits close to the ~500 LOC single-PR cap — see D0 for the split option.**

## Intent Verification

- [ ] Given a non-empty evidence list, `ProofSpine` renders a horizontally scrollable row of `ProofCard`s with a header reading "THE PROOF · {{count}} STEPS · SWIPE →" (count matches list length).
- [ ] Given a single evidence item, the row still renders (no special-cased "singular" layout diff beyond the header's pluralized count string).
- [ ] Given an item with `type: null`, its `ProofCard` renders with a neutral, type-agnostic token (no evidence-type color identity), shows a neutral bullet (`•`) glyph, renders **no** type tag — but is still tappable and still present in the list (not filtered out).
- [ ] Given an item with a known `type`, its `ProofCard`'s background/accent renders the corresponding `evidence-<type>-*` semantic token color — consistent per type, not decorative/random-by-index.
- [ ] Given an empty evidence array, `ProofSpine` renders one non-dismissible, non-interactive card with the exact honest-empty copy ("No evidence was attached to this goal — nothing to show in the gallery.") and no call-to-action, checkbox, or "add evidence" affordance anywhere in the tree.
- [ ] Tapping a non-empty `ProofCard` invokes `onCardPress(id)` with that item's `id` (the Storybook story wires this to a logging/action no-op — no real navigation in this PR).
- [ ] Switching the Storybook global theme toolbar through all 7 `ThemeName` values (`light-default`, `dark-default`, `light-highContrast`, `light-dyslexia`, `light-autismFriendly`, `light-lowVision`, `light-lowInfo`) never renders a literal hex string — verified by `grep -rn "#[0-9a-fA-F]\{3,6\}" ProofSpine.tsx ProofCard.tsx ProofSpine.styles.ts ProofCard.styles.ts` returning nothing outside comments.
- [ ] The new `evidence-*` token group resolves to a real (non-fallback) color for all **7 product themes** (`themeNames` in `src/themes/compose.ts:291-306`: `light-default`, `dark-default`, `light-highContrast`, `light-dyslexia`, `light-autismFriendly`, `light-lowVision`, `light-lowInfo` — variants are a light-only axis; dark ships solely as `dark-default`, so this is NOT a 14-theme matrix), and those 7 pass the existing WCAG AA contrast gate (`src/themes/__tests__/contrast.test.ts`, `contrastPairs.ts`) for every `evidence-<type>-bg`/`evidence-<type>-fg` pair, OR are explicitly ratcheted via `KNOWN_FAILURES` with a linked follow-up, mirroring the precedent already established for `journeyStepActive`/`journeyStepComplete` in that same file.
- [ ] `ProofSpine`'s outer container exposes `accessibilityRole="list"` with an `accessibilityLabel` announcing the count (mirrors `BadgeDetailScreen.tsx`'s `evidenceList` treatment); each `ProofCard` is a real `Pressable` with `accessibilityRole="button"` and a composed label of name + type — never one flattened parent node.
- [ ] Every `ProofCard` measures at least 44×44pt.
- [ ] `ProofSpine` is not imported by `BadgeDetailScreen.tsx` or any other screen at the end of this PR (grep for `ProofSpine` outside `src/components/ProofSpine/` and `.storybook*` returns nothing).

### Split option (numbers for Joe to decide)

Sizing is against the **7 product themes** (`themeNames`: `light-default`, `dark-default`, `light-highContrast`, `light-dyslexia`, `light-autismFriendly`, `light-lowVision`, `light-lowInfo`). Variants are a light-only axis — `dark` ships solely as `dark-default` — so, exactly like `journey-*`, this is a **base (light) + dark override + up to 5 light-variant overrides** shape, never a 14-way matrix authored in full.

**Option A — single PR (current plan):** ~430–480 LOC across ~20 files, 8 commits (see Implementation Plan). Mixes two concerns: a design-system token addition (touches `packages/design-tokens/`, a separate workspace package with its own build step and its own `CLAUDE.md` contract) and a presentational app component. Review surface spans two different "kinds" of change (token/contrast correctness vs. component/a11y correctness).

**Option B — split into two PRs:**

- **PR 1 (prerequisite): `evidence-*` token group.** Files: `packages/design-tokens/src/tokens/evidence.json` (new, ~60–90 lines for 7 types × 2 keys × DTCG wrapper — base **light** values only, mirrors `journey.json`'s 88 lines for a comparable key count), a **dark-mode override** in `packages/design-tokens/src/themes/dark.json` (only where the light value doesn't carry — mirrors `journey`'s 15-line dark block, ~0–15 lines), **light-variant overrides** in up to 5 of `packages/design-tokens/src/themes/{high-contrast,dyslexia-friendly,autism-friendly,low-vision,low-info}.json` (only where that variant's a11y contract requires a divergent value — modeled on `journey-*`'s sparse per-file override pattern: `high-contrast.json` has 7 journey lines, `autism-friendly`/`low-vision` have 4 each, `low-info` has 3 — realistically ~25–45 lines total across all 5; `light.json`/`large-text.json` get **zero** evidence overrides, matching `journey`'s existing 0-line entries in both), `packages/design-tokens/build-unistyles.js` (new `SEMANTIC_CATEGORIES` entry, ~20 lines), `src/themes/adapter.ts` (re-export block, ~10 lines, mirrors the existing 8-line `journey` block), `src/themes/variants.ts` (add `evidence?: EvidenceOverride` to the `VariantOverride` interface + `evidence: evidenceVariants.<name>` in each of the 5 ND-variant blocks — mirrors the existing 6 `journey` touch-points in this file, ~10 lines), `src/themes/compose.ts` (`ComposedTheme.evidence` field + base/override resolution block, ~10 lines, mirrors `compose.ts:151-156`), `src/themes/contrastPairs.ts` (7 new pairs, ~35 lines), plus `packages/design-tokens/build/unistyles/semanticColors.ts` regenerated (auto-generated, not hand-written, but counts toward the diff — modeled on the existing `journey`/`badgeReward` blocks, ~50–70 lines of generated output). **Total: ~9 files, ~170–210 LOC, 1–2 commits** (token JSON + build-script category; adapter/variants/compose/contrastPairs wiring). No app-visible behavior change — a pure plumbing PR, testable via the existing contrast test suite passing with the new pairs added, sized against 7 themes not 14.
- **PR 2 (this issue, consumer): `ProofSpine`/`ProofCard`.** Same file list as the component-only version of this plan (component, styles, stories, tests, i18n) — **~260 LOC across 11 files, 5 commits** — now importing `theme.evidence.*` from an already-shipped token group instead of adding it inline.

Option B keeps each PR under the cap with a single cohesive concern and gives the token group its own contrast-gate verification cycle before any component consumes it. Option A ships faster (no sequencing) but reviews two concerns at once and sits close to the cap. **Presented for Joe's call — not decided here.**
