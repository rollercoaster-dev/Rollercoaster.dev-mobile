## Issue Summary

**Title**: [Storybook] New Goal build step — replace bespoke rows with the Edit Goal step editor (EditGoalStepList)
**Type**: refactor (Storybook-scope; no live screen is wired to `NewGoalWizard` yet — `NewGoalModal.tsx` is still the old title-only modal, so this PR has zero user-facing runtime impact until #444 [Integrate])
**Complexity**: MEDIUM (issue's own `size:m` label; ~4 files touched, net line delta small per the issue's own estimate, but the interface rewrite + full test-suite rewrite make it a real cross-file rewire, not a trivial swap)
**Estimated Lines**: net small — deletes ~300–400 bespoke lines (build-row JSX/props/styles/tests **plus** the now-redundant `buildHeaderRow` header), adds ~150–250 wiring/story/test lines (incl. the ~16 copy passthrough props). ~4 files. **No change to the just-merged #489 `EditGoalStepList` file** (see D1).

## Resolved Decisions (2026-07-09, with Joe)

The five open questions from research were resolved before implementation:

1. **Header (D1)** → **Reuse EditGoalStepList's built-in header.** The wizard passes `stepsSectionLabel="Your steps"` and deletes its own `buildHeaderRow`. No new prop on #489.
2. **Add affordance (D3)** → **Adopt EditGoalStepList's typed-input + button add**, delete the wizard's `+ add another step` link. Dictated by the issue's "no fork" rule.
3. **Copy passthrough props (D4)** → **Add them now.** Re-expose EditGoalStepList's ~16 copy props on `NewGoalWizardProps`, matching EditGoalView and the wizard's own D5 "every string is a prop" contract.
4. **Delete-confirm story (D7)** → **Drop the frozen `BuildStepDeleteConfirm` story.** EditGoalStepList owns `pendingDelete` internally (no prop to seed it on mount), and `EditGoalView.stories.tsx` already reviews that modal for the same component.
5. **`dragScrollController` (D6)** → **Keep as a passthrough**, undefined in every story, mirroring EditGoalView exactly (consistent with the D4 "full contract" choice).

## Objective

Delete the wizard's second, weaker step editor (bespoke numbered `buildRowCard` rows, single-type evidence, no reorder) and swap the build step's body onto the already-extracted `EditGoalStepList`, so creating a goal and editing one use the identical row/list component **and the identical header treatment**. The wizard keeps only its own frame chrome (header band, progress bar, footer CTA). This also deletes the `BuildStep`/`EditGoalStep` data-model fork that #444 [Integrate] would otherwise have had to reconcile.
