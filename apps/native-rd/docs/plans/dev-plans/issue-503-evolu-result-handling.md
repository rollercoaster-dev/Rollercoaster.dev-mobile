# Development Plan: Issue #503

## Issue Summary

**Title**: [Foundation] Enforce Evolu Result handling at redesign mutation boundaries
**Type**: bug (foundation/refactor)
**Complexity**: LARGE (recommend a mid-plan split — see "Recommended split" below)
**Estimated Lines**: ~550-650 lines (impl + tests), before any split

## Intent Verification

- [x] When badge PNG storage or the `badge` DB insert/update fails after a bake attempt (`{ ok: false }`, no throw), `useCreateBadge` reports `status: "error"` — it never reaches `"done"` — and `completeGoal` is not called, so the goal stays active with no badge.
- [x] When Evolu rejects a goal title/description edit, a step create/update/delete, or a step reorder in `EditModeScreen` with `{ ok: false }`, the on-screen draft/list is left as the user last saw it and an error alert appears — the edit does not silently vanish.
- [x] When `deleteGoal` (Goals list or Edit Goal overflow) or `deleteBadge` (Badge Detail) returns `{ ok: false }`, the confirmation modal stays open, the screen does not navigate away, and the user sees an error — today both call sites close the modal and/or navigate unconditionally.
- [x] When `updateUserSettings` rejects a density or theme change, the Settings screen shows a toast and the previously-active option stays visually selected (no UI that looks "saved" while nothing persisted).
- [x] Every touched container/hook has at least one test that exercises the thrown-error path and one that exercises the `{ ok: false }` Result path.

## Objective

Every mutation caller in the redesigned Full Ride flows must treat Evolu's `{ ok: false }` Result exactly like a thrown exception: never fire a success-only UI side effect (clear a draft, close a destructive-confirmation modal, navigate away, advance a stage/card, or reach a terminal "done" state) unless the write actually succeeded. Where a call site currently has zero user-visible feedback on failure, add it using the app's existing feedback primitives (`Alert.alert` for blocking confirmations, `useToast` where a toast provider is in scope).

- [2026-07-21] Step 3 — `reorderSteps`/`reorderSubSteps` return **`void`**, not a `Result`. They delegate to `applyStepOrdinals`, which already checks `.ok` on each per-row `evolu.update` and **throws** an aggregate `Error` if any ordinal write fails. So `handleReorderSteps`/`handleReorderSubSteps` were left on their existing `try/catch` (already complete) rather than wrapped in `runEvoluMutation` — wrapping would be a type error (`runEvoluMutation` requires `() => Result`). The plan's Affected-Areas note that they were "missing the same check" was inaccurate for these two; every other EditModeScreen handler (`updateGoal`/`updateStep`/`deleteStep`/`createStep`/`createSubStep`/`deleteGoal`) does return a `Result` and was routed through `runEvoluMutation`.
- [2026-07-21] Step 5 — Changed `ThemeContextValue.setTheme` return type from `void` to `boolean` in `src/hooks/useTheme.ts` (not called out in the plan). `ThemeSwitcher` consumes `setTheme` via `useThemeContext()`, so the context type had to widen for the component to read the persist-success boolean. The outer-provider stub (throws → `never`) and the pre-Evolu Unistyles-only provider remain assignable.
- [2026-07-21] Step 5 — Hardened `ThemeSwitcher.stories.tsx` with a `ThemeProvider` + `ToastProvider` wrapper. The story was already provider-less (it used `useThemeContext()` with no `ThemeProvider` in Storybook's decorator); adding `useToast()` made a second provider mandatory. The wrapper supplies a working `setTheme` (applies via `UnistylesRuntime`, returns `true`) so the story renders and switches themes.
- [2026-07-21] Step 5 — Return-value contract for `useDensity.setDensity` / `useThemePersistence.setTheme`: return `false` **only** when an Evolu write was attempted and rejected (thrown or `{ ok: false }`). All non-write outcomes return `true` — no settings row yet (`!settings`), an unsupported theme name, and the in-flight rapid-tap drop — so the calling UI's error toast fires precisely on a real persistence failure and never on a debounced/guarded no-op.
