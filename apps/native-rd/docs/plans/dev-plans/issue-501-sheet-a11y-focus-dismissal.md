# Development Plan: Issue #501

## Issue Summary

**Title**: A11y: shared sheet focus, dismissal, restoration, and reduced motion
**Type**: bug (accessibility)
**Complexity**: MEDIUM
**Estimated Lines**: ~380 lines (implementation + tests)

## Intent Verification

- [x] Opening any `AnimatedSheet` consumer — the capture sheet, the build-step evidence sheet, the edit-goal evidence sheet, or (as of this issue) `BadgeShareSheet`, rewritten to compose `AnimatedSheet` instead of its own `Modal` — moves VoiceOver/TalkBack focus onto the sheet's title, and swiping past the last control in the sheet does not reach content behind it (Android: siblings carry `importantForAccessibility="no-hide-descendants"` while the sheet is open, in every screen where `AnimatedSheet` is actually mounted alongside other screen content; iOS: `accessibilityViewIsModal` already isolates).
- [x] `BadgeShareSheet` has a labelled, screen-reader-reachable Close control — `AnimatedSheet`'s own × button, inherited by composition rather than built bespoke — and closing it via that button, the backdrop, a row selection, or Android back all return focus to the "Share badge" CTA that opened it.
- [x] Closing any `AnimatedSheet` consumer (including the now-reused `BadgeShareSheet`) via the × button, backdrop tap, a grid/row selection, or Android hardware back returns focus to the control that opened it — the evidence chip, the "Add evidence" trigger, or the "Share badge" CTA — not to an arbitrary or unfocused element.
- [x] With the effective animation preference resolved to `"none"` (autism-friendly theme, OS reduce-motion, or user setting), none of the four sheets plays a slide animation. This is inherited automatically once `BadgeShareSheet` composes `<AnimatedSheet>`, which already resolves `getTimingConfig(animationPref, "normal")` to a 0ms duration — no `BadgeShareSheet`-specific animation code or test is needed; it's covered once, generically, by `AnimatedSheet.test.tsx` (Step 2).
- [x] Rapid open→close→open cycles (interrupting the exit animation) don't leave a stale focus-restoration timer that fires after the sheet has reopened, and don't regress the existing exit-content-retention behavior (sheet content stays rendered/visible through the fade/slide-out, per #493/#494 — now also true of `BadgeShareSheet` once it shares the same `AnimatedSheet` lifecycle).

## Objective

Give the app's shared in-tree bottom-sheet component, `AnimatedSheet` — used by the capture-evidence sheet, the New Goal Wizard's build-step evidence sheet, Edit Goal's evidence sheet, and, as of this issue, `BadgeShareSheet` (rewritten to compose it instead of maintaining a parallel bespoke `Modal` chrome) — a consistent, testable accessibility contract: focus enters the sheet on open, the rest of the screen is hidden from assistive tech while it's open wherever `AnimatedSheet` is mounted alongside other screen content, every sheet has a labelled Close, every dismissal path (button/backdrop/selection/Android back) restores focus to the control that opened it, and motion is fully suppressed when the effective animation preference is `"none"`.

## Discovery Log

- [2026-07-11] `focusAccessibilityRef` resolves the native tag **inside** the
  delayed callback, not at schedule time (Step 1 refinement). A sheet opened
  this render mounts its title on the _next_ commit, so `titleRef.current` is
  still null when the open effect runs — resolving eagerly made focus-on-open
  silently no-op on the false→true transition. Resolving lazily also means
  `{ current: null }` now schedules a (harmless) timer and returns a cancel fn
  rather than returning `undefined`; the unit tests were adjusted to match.
- [2026-07-11] `nativeEvent.target` is typed `string` by RN 0.85
  (`CoreEventTypes.d.ts`) though it's the numeric react tag at runtime — cast via
  `as unknown as number` in the EditGoalView/NewGoalWizard chip handlers. The
  handlers also read it through `event?.nativeEvent?.target` because
