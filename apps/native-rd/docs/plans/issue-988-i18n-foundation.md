## Goal

Stand up the i18n plumbing for `apps/native-rd` so subsequent tickets can migrate strings without re-litigating tooling. **No user-visible strings change in this PR.** Foundation is intentionally minimal — no string migrations and no user-facing locale changes — but it does install `expo-localization` (a native module, so a one-time native rebuild is required) and accepts the bare-string `"expo-localization"` plugin entry that `bunx expo install` auto-adds to `app.json`. The plugin entry is a no-op without `supportedLocales`; that config is deferred to #1029.

## Scope

## Architecture decisions

### Pseudo-locale activation: env var, dev-only

`selectSupportedLanguage` returns `"pseudo"` only when both:

1. `__DEV__` is `true` (production bundles **cannot** boot in pseudo), and
2. `process.env.EXPO_PUBLIC_I18N_PSEUDO === "true"`.

Rationale:

- Mirrors the existing `EXPO_PUBLIC_STORYBOOK_ENABLED` pattern (`index.ts:37`, `App.tsx:19`) — one fewer novel concept for reviewers.
- Keeps `selectSupportedLanguage` pure: function depends only on its argument plus values frozen at Metro start, so unit tests are deterministic without mocking AsyncStorage or rendering a Settings screen.
- Pseudo-locale snapshot tests (#1003) and Maestro pseudo runs can set the env var in their launch scripts.
- A user-facing Settings toggle (the eventual "real" mechanism for designers/QA) is intentionally deferred — it would pull UI scope into foundation. Add it alongside #1029 if needed.

### Import site: `apps/native-rd/index.ts`, not `App.tsx`

Add `require("./src/i18n")` immediately after the `initSentry()` call in `index.ts`, before the `App` require. Sentry is initialized first so any throw from i18next init is captured by telemetry; i18n still precedes the App require so screen modules never see an uninitialized `t()`. Reasons:

- `index.ts` is already the explicit-ordering boot file (its comments call this out). It runs before _both_ App.tsx and Storybook's entry.
- Synchronous side-effect import keeps `i18n` available to any module-level code that might call `t()` later. With `App.tsx` placement, any future top-level `t()` call in a screen file would race the init.
- `initAsync: false` + bundled JSON means init is synchronous; no need to await.

### Typed keys via traditional `CustomTypeOptions`

Per the issue body. The newer selector-mode (`enableSelector: true`) is more ergonomic for very deep keys but the team and review tooling expect dotted `t("a.b.c")` calls. Keep the standard pattern.

With `en.json` initially `{}`, `CustomTypeOptions["resources"]` types as `{ translation: {} }`, and `keyof {}` is `never` — so the typed `t()` rejects _any_ string key at compile time, not accepts any. The smoke test in `i18n.test.ts` works around this by casting `i18n.t` to `(key: string) => string` so it can exercise the runtime fallback for missing keys. Real key safety arrives once #992 populates `en.json`. **This is expected — call it out in the PR description so reviewers don't trip on the cast in the smoke test.**

### Synchronous init, no Suspense

```ts
i18n.use(initReactI18next).init({
  resources: { en: { translation: en }, pseudo: { translation: pseudo } },
  lng: selectSupportedLanguage(getLocales()),
  fallbackLng: "en",
  supportedLngs: ["en", "pseudo"],
  nonExplicitSupportedLngs: true,
  defaultNS: "translation",
  initAsync: false,
  interpolation: { escapeValue: false },
  react: { useSuspense: false },
});
```

- `nonExplicitSupportedLngs: true` lets future tags like `en-GB` resolve to `en` without an explicit listing.
- `interpolation.escapeValue: false` — React Native renders text directly, no HTML escaping needed (and would corrupt copy).
- `react.useSuspense: false` — no `<Suspense>` boundary required; matches existing render tree.

## Implementation notes

### `language.ts` shape

```ts
import type { Locale } from "expo-localization";

export type SupportedLanguage = "en" | "pseudo";

export function selectSupportedLanguage(
  locales: readonly Locale[] | null | undefined,
): SupportedLanguage {
  if (__DEV__ && process.env.EXPO_PUBLIC_I18N_PSEUDO === "true") {
    return "pseudo";
  }
  // First device locale wins; everything that isn't en-* falls back to en.
  // nonExplicitSupportedLngs on i18next handles the en-GB → en case at runtime;
  // this function only needs the coarse mapping for the initial lng.
  const code = locales?.[0]?.languageCode ?? "en";
  return code === "en" ? "en" : "en";
}
```

The `code === "en" ? "en" : "en"` is intentional and self-documenting: today there is only one supported user-facing locale. When #1029 lands German, the second branch becomes `"de"`. Leaving the ternary in foundation flags the extension point without adding dead branches.

### Pseudo-locale transform

The script must:

- Recursively walk the parsed JSON, only transforming string leaves.
- Skip substrings matching `/\{\{[^}]+\}\}/g` (i18next interpolation) — these are tokens, not display copy.
- Preserve i18next plural suffix keys (`_zero`, `_one`, `_other`) — they're map keys, not values.
- Apply the standard accent map (`a → à`, `e → ê`, `i → ï`, `o → ø`, `u → ü`, etc.) and pad ~40% by appending bracketed dots: `Hello → [Hêllø··]`. Brackets surface concatenation bugs.

For this PR the script's only observable output is `{}` → `{}`. Real coverage arrives when #992 populates `en.json`. Smoke-test the script with one fixture in `scripts/__tests__/` if it pays off — otherwise rely on manual verification this PR.

### Jest mock contract

```ts
// src/__tests__/mocks/expo-localization.ts
export const getLocales = () => [
  {
    languageTag: "en-US",
    languageCode: "en",
    regionCode: "US",
    textDirection: "ltr",
    digitGroupingSeparator: ",",
    decimalSeparator: ".",
    measurementSystem: "us",
    currencyCode: "USD",
    currencySymbol: "$",
    temperatureUnit: "fahrenheit",

];
export const useLocales = getLocales;
```

Match `expo-localization`'s `Locale` shape so `selectSupportedLanguage` compiles against the real type. Wire in `jest.config.js`'s `moduleNameMapper` under `"^expo-localization$"`.
