# Internationalization (i18n)

How to use translations in `native-rd`. Read this before migrating literal strings to `t()` calls or adding new user-facing copy.

The foundation lives at `src/i18n/`. Architecture rationale and milestone planning are in [`docs/research/native-rd-translations-research.md`](research/native-rd-translations-research.md) and [`/docs/plans/active/milestone-30-i18n.md`](../../../docs/plans/active/milestone-30-i18n.md).

---

## Stack

| Layer                   | Library             | Purpose                                                                  |
| ----------------------- | ------------------- | ------------------------------------------------------------------------ |
| Device locale discovery | `expo-localization` | Reads the user's ordered locale preferences via `getLocales()`.          |
| Translation engine      | `i18next`           | Resource bundles, fallbacks, interpolation, plurals, language switching. |
| React integration       | `react-i18next`     | `useTranslation()` hook and `<Trans>` component.                         |

All three are initialized synchronously at app boot in `index.ts` (after `initSentry()`, before the App require), so `t()` is available to any module-level or screen-level code that runs after that point. Sentry comes first so that any throw from i18next init reaches telemetry.

## File layout

```text
src/i18n/
├── index.ts              # i18next initialization (do not import elsewhere)
├── language.ts           # selectSupportedLanguage(locales) — pure, tested
├── i18next.d.ts          # CustomTypeOptions augmentation for typed t() keys
├── pseudoTransform.ts    # Pure pseudo-locale transform (unit-tested)
├── resources/
│   ├── en.json           # Canonical English — the source of truth
│   └── pseudo.json       # Generated from en.json; do not edit by hand
└── __tests__/            # language, i18n, pseudo-locale tests

scripts/
└── generate-pseudo-locale.ts   # Regenerates resources/pseudo.json from en.json
```

## Using translations in a component

The basic pattern:

```tsx
import { useTranslation } from "react-i18next";

export function SaveButton({ onPress }: Props) {
  const { t } = useTranslation();
  return (
    <Button
      label={t("common.actions.save")}
      accessibilityLabel={t("common.actions.save")}
      onPress={onPress}
    />
  );
}
```

Key calls:

- **`useTranslation()`** — returns `{ t, i18n }`. Call once per component.
- **`t("dotted.key.path")`** — looks up a string. Returns the key itself if missing (does not throw).
- **`i18n.changeLanguage(lng)`** — async; usually only needed in tests or a future Settings toggle.

### Interpolation

```tsx
t("screens.goals.welcomeUser", { name: user.name });
// en.json: { "screens": { "goals": { "welcomeUser": "Welcome back, {{name}}!" } } }
```

The `{{name}}` token is a placeholder, not display text. The pseudo-locale generator preserves these tokens verbatim — never accent or pad them.

### Plurals

i18next picks the form by the `count` value (uses `Intl.PluralRules` under Hermes):

```jsonc
{
  "common": {
    "evidence": {
      "item_zero": "+ add evidence",
      "item_one": "{{count}} item",
      "item_other": "{{count}} items",
    },
  },
}
```

```tsx
t("common.evidence.item", { count: items.length });
```

Always include `_other` (the fallback). `_zero` is optional — i18next falls back to `_other` for zero unless `_zero` is present.

### Accessibility strings are user-facing copy

`accessibilityLabel`, `accessibilityHint`, screen-reader announcements — all of them go through `t()`. They are read aloud to users with screen readers; they need to be translated for the same reason visible labels do.

```tsx
<Pressable
  accessibilityRole="button"
  accessibilityLabel={t("screens.goals.createButton.label")}
  accessibilityHint={t("screens.goals.createButton.hint")}
/>
```

## Key naming conventions

Group by **user workflow**, not by component file name. Keys are stable contracts with translators and outlive components.

```jsonc
{
  "common": {
    "actions": { "save": "Save", "cancel": "Cancel", "delete": "Delete" },
  },
  "evidenceTypes": {
    "photo": { "label": "Take Photo", "shortLabel": "Photo" },
  },
  "screens": {
    "goals": {
      "title": "Goals",
      "emptyTitle": "No goals yet",
      "createAccessibilityLabel": "Create new goal",
    },
  },
}
```

Rules:

- **Dotted paths, lowerCamelCase segments.** `screens.goals.title`, not `Screens_Goals_TITLE` or `screens-goals-title`.
- **Reuse `common.*` for cross-cutting copy.** If you're adding `goalsScreen.saveButton` and there's already `common.actions.save`, use the existing key.
- **Semantic over visual.** `screens.goals.emptyTitle` is fine; `screens.goals.bigBoldText` is not.
- **No component-name prefixes.** `pageButton.save` ages badly when the button moves screens. `common.actions.save` doesn't.

## What NOT to translate

User-generated content and machine identifiers never go through `t()`:

- Goal titles, step titles, evidence captions — anything the user typed.
- Badge names/descriptions when those came from imported Open Badges data.
- URLs, file names, credential payloads, IDs, hashes.
- Date/time strings that are formatted via `Intl` rather than templated — those get locale-aware formatting (see #989), not translation.

If you find yourself wanting to translate the value `goal.title`, you're in the wrong place — the field itself is user input.

## Adding a new translatable string — workflow

1. **Pick a key.** Group by workflow. Reuse `common.*` if the copy already exists.
2. **Add the English value to `src/i18n/resources/en.json`.**
3. **Call `t("your.new.key")` from the component.**
4. **Regenerate the pseudo locale**: `bun run gen:pseudo` from `apps/native-rd/`.
5. **Commit `en.json`, `pseudo.json`, and the component change together.** Reviewers should be able to see the key, the value, and the consumer in one diff.
6. **Type-check + test**: `bun run type-check && bun run test`. The typed `CustomTypeOptions` will fail the build if you typo'd the key path.

When the eventual second locale (German, per #1029) ships, you'll add a `de.json` with the same key structure. Until then, only `en.json` is hand-maintained.

## Don't concatenate translated fragments

```tsx
// Wrong — translators can't reorder, plural agreement breaks across languages
<Text>{t("goals.youHave")} {count} {t("goals.goalsLeft")}</Text>

// Right — one full sentence with interpolation + plural keys
<Text>{t("goals.remaining", { count })}</Text>
```

Languages reorder words differently than English does. Sentence fragments smashed together with JSX produce ungrammatical output the moment a real translator gets involved. The pseudo locale will surface this immediately — you'll see `[…][…]` (two bracket pairs) instead of one.

## Pseudo locale

The pseudo locale is a development/QA tool, not a user-facing language. It's a synthetic locale generated from `en.json` that:

- Accents every ASCII letter (`Save → Šàṽê`)
- Surrounds every string with brackets (`[Šàṽê]`)
- Pads ~40% to mimic German/Finnish length

Bugs it catches:

| Bug                               | How pseudo surfaces it                                                  |
| --------------------------------- | ----------------------------------------------------------------------- |
| Forgot a `t()` call               | One label on screen is plain English while everything else is bracketed |
| Concatenation                     | Two adjacent bracket pairs `[…][…]` instead of one wrapping pair        |
| Text overflow in fixed containers | Padded copy clips or wraps in obvious ways                              |
| Encoding/font issues              | Mojibake or `?` boxes where accents should be                           |

### Enabling pseudo

Add to `apps/native-rd/.env.local`:

```sh
EXPO_PUBLIC_I18N_PSEUDO=true
```

Then **fully restart Metro** (Ctrl+C, then `bun run ios`). A reload alone won't pick up the change — `EXPO_PUBLIC_*` vars are read at bundle time, not at runtime.

### Safety

- `selectSupportedLanguage()` requires both `__DEV__` and the env var. Production bundles cannot resolve to pseudo regardless of env.
- `pseudo.json` is generated, never hand-edited. Re-run `bun run gen:pseudo` after any change to `en.json`.

### When to use it

- Pulling a colleague's i18n migration PR — open every changed screen under pseudo.
- Designing a new screen — check that copy doesn't truncate before a real translator delivers German.
- Writing snapshot tests for representative screens (issue #1003) — switch i18next to `pseudo` in `beforeEach`.

## Testing components that use `t()`

Most tests should keep asserting against English copy — that's what the device shows users today. Don't duplicate every screen test per locale.

```tsx
import { renderWithProviders } from "@/__tests__/test-utils";

test("save button is labeled", () => {
  const { getByText } = renderWithProviders(<SaveButton onPress={noop} />);
  expect(getByText("Save")).toBeTruthy();
});
```

Two specific cases that _do_ need locale-aware tests:

1. **Pseudo-locale snapshot tests** (issue #1003) — confirm representative screens render _some_ string for every label, regardless of language.
2. **Pluralization** — explicit `t("common.evidence.item", { count })` cases at `0`, `1`, `2` to confirm the right form fires.

If you're worried about test brittleness from copy changes, prefer accessibility-role queries over text queries:

```tsx
// More resilient to copy tweaks
getByRole("button", { name: t("common.actions.save") });
```

## Native strings — out of scope here

iOS permission dialogs, the app display name, and other strings rendered from `app.json` are **not** routed through i18next. They use Expo's native localization files (`apps/native-rd/locales/*.json`), wired by adding `supportedLocales` to the `expo-localization` config plugin in `app.json`. That work is deferred to issue #1029 (German first-test path) — don't add it ad hoc.

## Pre-PR checklist

When you've migrated strings in a PR, check:

- [ ] Every literal in `screens/` and `components/` you touched is now `t()`-wrapped.
- [ ] New keys are in `en.json`, grouped semantically.
- [ ] `bun run gen:pseudo` was run; `pseudo.json` is up to date.
- [ ] Accessibility labels and hints on touched components are translated too.
- [ ] No translated fragment concatenations.
- [ ] Tested visually under `EXPO_PUBLIC_I18N_PSEUDO=true`: no plain-English copy, no `[…][…]` patterns.
- [ ] `bun run type-check` and `bun run test` pass.
- [ ] If you added stable `testID`s to decouple from copy churn, they follow the conventions in #991.
