# Internationalization (i18n)

How to use translations in `native-rd`. Read this before migrating literal strings to `t()` calls or adding new user-facing copy.

The foundation lives at `src/i18n/`. Architecture rationale and milestone planning are in [`docs/research/native-rd-translations-research.md`](research/native-rd-translations-research.md) and [`docs/plans/milestone-3-i18n.md`](plans/milestone-3-i18n.md).

---

## Stack

| Layer                   | Library             | Purpose                                                                  |
| ----------------------- | ------------------- | ------------------------------------------------------------------------ |
| Device locale discovery | `expo-localization` | Reads the user's ordered locale preferences via `getLocales()`.          |
| Translation engine      | `i18next`           | Resource bundles, fallbacks, interpolation, plurals, language switching. |
| React integration       | `react-i18next`     | `useTranslation()` hook and `<Trans>` component.                         |

All three are initialized synchronously at app boot in `index.ts` (after `initSentry()`, before the App require), so `t()` is available to any module-level or screen-level code that runs after that point. Sentry comes first so that any throw from i18next init reaches telemetry.


**Why namespaces?** One JSON per workflow area lets parallel screen-migration PRs touch disjoint files — no merge conflicts on a shared `en.json`. The default namespace is `common` (shared labels). Screen-specific keys live in their own namespace and are accessed via `useTranslation("welcome")` or the `welcome:title` colon syntax.

## Using translations in a component

The basic pattern:

```tsx
import { useTranslation } from "react-i18next";

// Shared labels — default namespace ("common"), no explicit useTranslation arg.
export function SaveButton({ onPress }: Props) {
  const { t } = useTranslation();
  return (
    <Button
      label={t("actions.save")}
      accessibilityLabel={t("actions.save")}
      onPress={onPress}
    />
  );
}

// Screen-specific labels — name the namespace and scope t() to it.
export function WelcomeHeader() {
  const { t } = useTranslation("welcome");
  return <Heading>{t("title")}</Heading>;
}

// Cross-namespace lookup from anywhere — use the colon syntax.
const goalsTitle = t("goals:title");
```

Key calls:

- **`useTranslation()`** — returns `{ t, i18n }` scoped to the default namespace (`common`). Call once per component.
- **`useTranslation("welcome")`** — same, but `t()` resolves keys inside the `welcome` namespace.
- **`t("dotted.key.path")`** — looks up a string in the current namespace. Returns the key itself if missing (does not throw).
- **`t("ns:dotted.key.path")`** — colon prefix overrides the namespace for one call. Useful for one-off cross-namespace references.
- **`i18n.changeLanguage(lng)`** — async; usually only needed in tests or a future Settings toggle.

### Cross-namespace lookups

Three forms exist in the codebase. They're not interchangeable — the right one depends on whether you're at a call site or inside a helper.

**At a component call site — use colon prefix (or no prefix when already in default ns):**

```tsx
// ✅ Scoped hook, colon prefix to reach `common`
const { t } = useTranslation(["welcome", "common"]);
<Button label={t("cta.getStarted")} />              // welcome ns
<Text>{t("common:theme.preview.title")}</Text>      // common ns, colon prefix

// ✅ Bare hook (defaultNS = common), no prefix needed for common keys
const { t } = useTranslation();
<View accessibilityLabel={t("a11y.evidenceTypeOptions")} />

// ❌ Avoid — inline options bag at call sites
<Text>{t("theme.preview.title", { ns: "common" })}</Text>
//                                ^^^^^^^^^^^^^ use colon prefix (or drop, if default)
```

The **array form** `useTranslation(["welcome", "common"])` is required when you use the colon prefix from a screen-scoped hook, because i18next's `CustomTypeOptions` typing only allows keys from namespaces you declared. Single-namespace screens that don't reach into `common` keep `useTranslation("settings")`.

**Inside helper functions — use options bag (this is the one exception):**

```ts
// src/i18n/labels.ts
import type { TFunction } from "i18next";

const COMMON_NS = { ns: "common" } as const;

export function evidenceLabel(t: TFunction, type: EvidenceTypeValue): string {
  return t(`evidenceTypes.${type}.label`, COMMON_NS);
  //                                       ^^^^^^^^^^ required — the helper
  //  can't know what namespace the caller's `t` is scoped to, so it forces
  //  the lookup to `common` via the options bag.
}
```

Colon prefix doesn't work inside helpers because the caller's `t` may be typed against a namespace that doesn't include `common`, and `t("common:key")` would fail strict typing. The options-bag bypasses that.

**Helper threshold:** extract into `src/i18n/labels.ts` when the same key (or template-built key family like `evidenceTypes.<type>.*`) appears in ≥2 components. Single-consumer lookups stay inline at the call site.

### Interpolation

```tsx
t("screens.goals.welcomeUser", { name: user.name });
// resources/en/goals.json: { "welcomeUser": "Welcome back, {{name}}!" }, then useTranslation("goals")
```

The `{{name}}` token is a placeholder, not display text. The pseudo-locale generator preserves these tokens verbatim — never accent or pad them.

### Plurals

i18next picks the form by the `count` value (uses `Intl.PluralRules` under Hermes):

```jsonc
{
  "common": {
    "evidence": {
      "item_zero": "+ add evidence",
      "item_one": "{{count}} item",
      "item_other": "{{count}} items",
    },
  },
}
```

```tsx
t("common.evidence.item", { count: items.length });
```

Always include `_other` (the fallback). `_zero` is optional — i18next falls back to `_other` for zero unless `_zero` is present.

### Accessibility strings are user-facing copy

`accessibilityLabel`, `accessibilityHint`, screen-reader announcements — all of them go through `t()`. They are read aloud to users with screen readers; they need to be translated for the same reason visible labels do.

```tsx
<Pressable
  accessibilityRole="button"
  accessibilityLabel={t("screens.goals.createButton.label")}
  accessibilityHint={t("screens.goals.createButton.hint")}
/>
```

## Key naming conventions

One JSON per namespace; keys are top-level inside that namespace. Group by **user workflow**, not by component file name. Keys are stable contracts with translators and outlive components.

```jsonc
// resources/en/common.json
{
  "actions": { "save": "Save", "cancel": "Cancel", "delete": "Delete" },
  "evidenceTypes": {
    "photo": { "label": "Take Photo", "shortLabel": "Photo" }
  }
}

// resources/en/goals.json
{
  "title": "Goals",
  "emptyTitle": "No goals yet",
  "createAccessibilityLabel": "Create new goal"
}
```

```tsx
// Consumer pattern
const { t } = useTranslation("goals");
t("title"); // → "Goals" (goals namespace)
t("common:actions.save"); // → "Save"  (cross-ns colon prefix)
```

Rules:

- **One JSON per namespace.** Don't nest `screens.goals.*` inside a single mega-file — that was the pre-PR-#108 shape and is no longer the convention.
- **Dotted paths, lowerCamelCase segments.** `hero.title`, not `Hero_TITLE` or `hero-title`.
- **Reuse `common.*` for cross-cutting copy.** If you're adding `goals:saveButton` and there's already `common:actions.save`, use the existing key.
- **Semantic over visual.** `emptyTitle` is fine; `bigBoldText` is not.
- **No component-name prefixes.** `pageButton.save` ages badly when the button moves screens. `common:actions.save` doesn't.

## What NOT to translate

User-generated content and machine identifiers never go through `t()`:

- Goal titles, step titles, evidence captions — anything the user typed.
- Badge names/descriptions when those came from imported Open Badges data.
- URLs, file names, credential payloads, IDs, hashes.
- Date/time strings that are formatted via `Intl` rather than templated — those get locale-aware formatting (see #62), not translation.

If you find yourself wanting to translate the value `goal.title`, you're in the wrong place — the field itself is user input.

## Adding a new translatable string — workflow

1. **Pick a namespace.** Shared/cross-cutting copy goes in `common`. Screen-scoped copy goes in that screen's namespace (`welcome`, `goals`, `focusMode`, etc.).
2. **Pick a key** inside the namespace. Group by workflow. Reuse `common.*` if the copy already exists rather than duplicating into a screen namespace.
3. **Add the English value to `src/i18n/resources/en/<ns>.json`.**
4. **Call `t("your.new.key")` from the component** (with `useTranslation("<ns>")` for screen-scoped keys, or `useTranslation()` for default-namespace shared labels).
5. **Regenerate the pseudo locale**: `bun run gen:pseudo` from `apps/native-rd/`. The script auto-discovers files in `resources/en/` — no script changes needed when adding namespaces.
6. **Commit `resources/en/<ns>.json`, `resources/pseudo/<ns>.json`, and the component change together.** Reviewers should be able to see the key, the value, and the consumer in one diff.
7. **Type-check + test**: `bun run type-check && bun run test`. The typed `CustomTypeOptions` will fail the build if you typo'd the key path.

**Adding an entirely new namespace?** Create `resources/en/<ns>.json` (= `{}`) and `resources/pseudo/<ns>.json` (= `{}`), then register it in `src/i18n/index.ts` (NAMESPACES + resources bundles) and `src/i18n/i18next.d.ts`. The namespace-registration test in `__tests__/i18n.test.ts` will catch it if any of those four pieces drift.

When the German first-test path ships, add `de.json` with the same key structure. The first German batch may be generated, but #76 does not close until a native speaker has reviewed the German resources and native locale strings.

## Drift guards for dynamic keys

When you build keys from a runtime value — typically a closed TS union — the strict typing in `i18next.d.ts` only narrows the literal-template result. It does **not** check that every union member has a matching JSON entry. If the union and JSON drift apart, production renders the key path verbatim because `missingKeyHandler` is `__DEV__`-gated (see `src/i18n/index.ts:109-130`).

```tsx
// densityOptions: readonly { id: DensityLevel }[]
//   where DensityLevel = "compact" | "default" | "comfortable"
densityOptions.map((opt) => (
  <SettingsRow
    label={t(`density.options.${opt.id}.label`)}
    //         ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^ template literal — no per-id check
  />
));
```

If a union member is renamed without updating the JSON, this lookup silently renders `density.options.cozy.label` as UI in prod with no telemetry. Closed by `src/i18n/__tests__/option-key-parity.test.ts`, which asserts both directions:

1. Every union id → corresponding JSON key resolves (forward).
2. JSON keyset under the subtree → equals `union.map(o => o.id)` (catches orphan entries left after a rename).

**If you introduce a new template-literal `t()` call** with a runtime-built key, add the same pair of assertions to `option-key-parity.test.ts` — pattern lives there. Forward-only coverage is a known regression net hole.

## Don't concatenate translated fragments

```tsx
// Wrong — translators can't reorder, plural agreement breaks across languages
<Text>{t("goals.youHave")} {count} {t("goals.goalsLeft")}</Text>

// Right — one full sentence with interpolation + plural keys
<Text>{t("goals.remaining", { count })}</Text>
```

Languages reorder words differently than English does. Sentence fragments smashed together with JSX produce ungrammatical output the moment a real translator gets involved. The pseudo locale will surface this immediately — you'll see `[…][…]` (two bracket pairs) instead of one.

## Pseudo locale

The pseudo locale is a development/QA tool, not a user-facing language. It's a synthetic locale generated from every `resources/en/<ns>.json` (one pseudo JSON per namespace, mirroring the en layout) that:

- Accents every ASCII letter (`Save → Šàṽê`)
- Surrounds every string with brackets (`[Šàṽê]`)
- Pads ~40% to mimic German/Finnish length

Bugs it catches:

| Bug                               | How pseudo surfaces it                                                  |
| --------------------------------- | ----------------------------------------------------------------------- |
| Forgot a `t()` call               | One label on screen is plain English while everything else is bracketed |
| Concatenation                     | Two adjacent bracket pairs `[…][…]` instead of one wrapping pair        |
| Text overflow in fixed containers | Padded copy clips or wraps in obvious ways                              |
| Encoding/font issues              | Mojibake or `?` boxes where accents should be                           |

### Enabling pseudo

Add to `apps/native-rd/.env.local`:

```sh
EXPO_PUBLIC_I18N_PSEUDO=true
```

Then **fully restart Metro** (Ctrl+C, then `bun run ios`). A reload alone won't pick up the change — `EXPO_PUBLIC_*` vars are read at bundle time, not at runtime.

### Safety

- `selectSupportedLanguage()` requires both `__DEV__` and the env var. Production bundles cannot resolve to pseudo regardless of env.
- Files under `resources/pseudo/` are generated, never hand-edited. Re-run `bun run gen:pseudo` after any change to any `resources/en/<ns>.json`.

### When to use it

- Pulling a colleague's i18n migration PR — open every changed screen under pseudo.
- Designing a new screen — check that copy doesn't truncate before a real translator delivers German.
- Writing snapshot tests for representative screens (issue #75) — switch i18next to `pseudo` in `beforeEach`.

## Testing components that use `t()`

Two assertion shapes, both correct, picked by what the test is proving:

```tsx
import { renderWithProviders, screen } from "@/__tests__/test-utils";
import { i18n } from "@/i18n";

// (a) Look up via i18n.t — preferred. Survives copy tweaks; key path
//     is the assertion's contract, not the English string.
test("save button is labeled", () => {
  renderWithProviders(<SaveButton onPress={noop} />);
  expect(screen.getByText(i18n.t("common:actions.save"))).toBeOnTheScreen();
});

// (b) Role-based query — strongest when the assertion is about interaction.
test("save button is pressable", () => {
  renderWithProviders(<SaveButton onPress={noop} />);
  fireEvent.press(
    screen.getByRole("button", { name: i18n.t("common:actions.save") }),
  );
});
```

**Why not hardcode the English literal?** Two reasons:

1. The same assertion would still pass if a developer reverted the `t()` call to a hardcoded `<Text>Save</Text>` — in en mode, `i18n.t()` returns the same English string a literal would. That regression is invisible to en-mode tests.
2. When you rename or rephrase copy, every literal-string test in the file fails on something orthogonal to what the test is actually checking. The key path is stable; the copy isn't.

### Pseudo-render smoke tests

For every newly-migrated screen, add a `describe("pseudo locale")` block with a `beforeEach`/`afterEach` that switches `i18n` to `pseudo` and back. Assert one or more representative keys render bracketed text:

```tsx
describe("pseudo locale", () => {
  afterEach(async () => {
    if (i18n.language !== "en") await i18n.changeLanguage("en");
  });

  it.each([
    "welcome:hero.title",
    "welcome:intro.body1",
    "welcome:cta.getStarted",
  ] as const)("renders %s as bracketed copy under pseudo", async (key) => {
    await i18n.changeLanguage("pseudo");
    renderWithProviders(<WelcomeScreen onGetStarted={jest.fn()} />);
    const pseudo = i18n.t(key);
    expect(pseudo.startsWith("[")).toBe(true);
    expect(screen.getByText(pseudo)).toBeOnTheScreen();
  });
});
```

This is the only test shape that catches a reverted-to-hardcoded literal — under pseudo, `i18n.t()` returns bracketed text that a `<Text>Welcome to your ride.</Text>` literal won't match. Cover 3–5 representative keys per screen (title / body / CTA / error). One key per screen is too thin: a developer reverting every string except the one asserted key would still pass.

For TextInput placeholders, use `screen.getByPlaceholderText(pseudo)` — placeholders are a prop, not a Text node.

### Other locale-aware test cases

1. **Pluralization** — explicit `t("common:evidence.item", { count })` cases at `0`, `1`, `2` to confirm the right form fires (when #66 unblocks plurals).
2. **Pseudo-locale snapshot tests** (issue #75) — confirm representative screens render _some_ string for every label.

## Native strings

iOS permission dialogs, the app display name, and other strings rendered from `app.json` are **not** routed through i18next. They use Expo's native localization files (`apps/native-rd/locales/*.json`), wired by adding `supportedLocales` to the `expo-localization` config plugin in `app.json`.

For German first-test readiness, native strings are required. Do this through #61 so testers do not see a mix of translated app copy and English system permission dialogs.

## Pre-PR checklist

When you've migrated strings in a PR, check:

- [ ] Every literal in `screens/` and `components/` you touched is now `t()`-wrapped.
- [ ] New keys are in the correct `resources/en/<ns>.json`, grouped semantically.
- [ ] `bun run gen:pseudo` was run; `resources/pseudo/<ns>.json` files are up to date.
- [ ] Accessibility labels and hints on touched components are translated too.
- [ ] No translated fragment concatenations.
- [ ] Inline cross-namespace lookups use the colon prefix (`t("common:foo")`), not the options bag. Options bag is only for helpers in `labels.ts` that take a generic `TFunction`. Use a `labels.ts` helper if ≥2 components share the lookup.
- [ ] Template-literal `t()` calls have matching pairs of assertions in `src/i18n/__tests__/option-key-parity.test.ts`.
- [ ] Migrated screens have a `describe("pseudo locale")` block covering 3–5 representative keys.
- [ ] Test assertions use `i18n.t("ns:key")` lookups, not hardcoded English literals.
- [ ] Tested visually under `EXPO_PUBLIC_I18N_PSEUDO=true`: no plain-English copy, no `[…][…]` patterns.
- [ ] `bun run type-check` and `bun run test` pass.
- [ ] If you added stable `testID`s to decouple from copy churn, they follow the conventions in #64.
